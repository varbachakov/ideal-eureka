# Инструкции Для ИИ По Проекту Mobile Yarmy

## Контекст Проекта

Этот репозиторий - мобильное приложение нового направления Ярми для коротких реальных видеоисторий.

Важно: это не браузерный/веб-проект Ярми. Веб-проект живет отдельно. Этот проект отвечает за мобильную съемку реальных коротких видео. Стратегически Yarmy выносит идеологию сторис в браузерный мир, но этот репозиторий отвечает за то, чтобы история была снята реально, через мобильную камеру.

Главная идея продукта: история, реализм и присутствие. Пользователь показывает всем, где он находится, что делает и что происходит вокруг него прямо сейчас. Видео в Yarmy - это не импортированный файл для публикации, а событие, записанное камерой внутри приложения.

Продуктовый ориентир после изучения Telegram Stories: сторис работают как легкий слой присутствия. Они дают человеку быстро показать момент, не ломая основной сценарий продукта; автор управляет аудиторией; зрители могут быстро отреагировать или ответить; хорошие моменты могут жить в профиле дольше обычной истории. Но Telegram Stories позволяют выбрать медиа из галереи и использовать редактор. Yarmy берет философию сторис, но делает ее строже: только настоящий контент, снятый здесь и сейчас через камеру приложения.

Yarmy берет из логики сторис:

- первый экран и главный жест продукта должны вести к съемке;
- публикация должна быть быстрой и не терять уже снятый момент;
- базовый смысл публикации - показать живую ситуацию: где ты, что делаешь, что видишь;
- перед публикацией пользователь должен понимать, что произойдет с видео;
- просмотр истории должен позволять зрителям быстро реагировать на настоящий момент;
- в будущем социальная механика должна быть близка к Telegram Stories по простоте реакций и к TikTok по развлекательной легкости;
- позже можно добавить настройки видимости, счетчики просмотров, лайки, комментарии, ответы и профильные подборки;
- временность или постоянное хранение истории - продуктовый выбор, который должен быть явным для автора.

Yarmy сознательно отличается от Telegram Stories:

- видео записывается только через встроенную камеру приложения;
- загрузка из галереи запрещена, потому что она разрушает обещание настоящего контента;
- фильтры, AI-генерация, внешний монтаж и импортированные файлы запрещены;
- ценность продукта - доказательство происхождения: ролик был снят в приложении;
- медиа-редактор не должен становиться главным продуктом;
- сторис-механики не должны появляться раньше стабильного capture-flow;
- сайт/веб-приложение позже может показывать ленту, но этот репозиторий отвечает за мобильный capture-flow.

Основной пользовательский сценарий:

1. Пользователь открывает приложение.
2. Камера открывается максимально быстро.
3. Пользователь снимает реальное видео.
4. После записи видит предпросмотр.
5. Пользователь может обрезать начало/конец.
6. Затем выбирает: опубликовать или удалить.
7. Если пользователь не авторизован, вход запрашивается только перед публикацией, чтобы момент съемки не был потерян.

## Правила MVP

- Длина видео: минимум 3 секунды, максимум 60 секунд.
- Запись: один непрерывный дубль.
- Паузы во время записи нет.
- Выбора видео из галереи нет.
- Фильтров нет.
- AI-обработки в capture-flow нет.
- Счетчики просмотров, лайки и комментарии не входят в MVP, но являются целевым направлением развития историй.
- Обрезка начала/конца разрешена: для продукта важен факт съемки через камеру, а не точная сырая длительность.
- В MVP делаем обычную запись с одной камеры.
- Dual-camera в MVP не делаем.

Будущее направление камеры:

- MVP: Flutter `camera` plugin.
- Позже при необходимости: собственные нативные camera-модули.
- iOS: Swift + AVFoundation / AVCaptureSession.
- Android: Kotlin + CameraX / Camera2.
- Flutter должен хранить общий UI и продуктовый сценарий.
- Нативный код камеры должен быть изолирован за отдельной абстракцией, чтобы его можно было заменить без переписывания всего приложения.

## Архитектурное Направление

С самого начала держим проект модульным. Камеру нужно проектировать так, чтобы позже ее можно было заменить на нативную реализацию.

Рекомендуемая структура:

```text
lib/
  app/
  features/
    camera_capture/
  shared/
```

Правила:

- UI камеры и реализация камеры должны быть разделены.
- Не размазывать прямые вызовы `camera` plugin по всему приложению.
- Продуктовые сценарии складывать в feature-срезы.
- Переиспользуемые утилиты и базовые компоненты держать в `shared/`.
- На старте писать просто: не усложнять архитектуру раньше времени.

## Правила Разработки

- Все продуктовые UI-тексты должны быть на русском.
- Первый экран приложения должен быть практическим, не лендингом.
- Целевой первый экран - камера.
- Не добавлять социальную ленту до рабочего MVP съемки.
- Не копировать Telegram Stories целиком: использовать их как продуктовый ориентир для скорости, приватности, реакций и легкого присутствия.
- Не добавлять города, геолокацию, "рядом" и фильтры в MVP.
- Не добавлять просмотры, лайки и комментарии до рабочего MVP съемки, но проектировать продукт так, чтобы их можно было добавить позже.
- Не делать dual-camera до стабильной single-camera записи.
- Не добавлять загрузку из галереи даже как debug/hidden-функцию без явного запроса. Если такая возможность когда-либо обсуждается, сначала нужно явно зафиксировать, что она противоречит принципу настоящего контента.

## Важно Для Будущих ИИ-Чатов

При открытии этого репозитория в новом чате не считать его браузерной Ярми. Это мобильное приложение.

Стратегическая цель - не сделать второй TikTok и не сделать клон Telegram Stories. Yarmy переносит идеологию сторис в браузер: история как живое публичное присутствие, реализм, "где я и что я делаю", быстрые реакции зрителей на настоящий момент. По реакции Yarmy стремится к простоте Telegram Stories, по развлекательности - к легкости TikTok, но без отказа от настоящего контента. Отличие мобильного приложения: реальные видеоистории, снятые внутри приложения, без загрузки из галереи и без генеративного/монтажного слоя в capture-flow.

Перед изменением кода нужно изучить текущие файлы проекта и следовать Flutter-конвенциям, которые уже есть в репозитории.
