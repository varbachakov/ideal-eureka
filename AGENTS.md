# Инструкции Для ИИ По Проекту Mobile Yarmy

Главный продуктовый источник: `PRODUCT_CONCEPT.md`.
Короткий источник правил разработки: `PROJECT_RULES.md`.
Перед началом разработки сверяйся с ними, затем изучай текущие файлы проекта.

## Контекст Проекта

Этот репозиторий - мобильное приложение Ярми и первый capture-клиент платформы подтвержденной реальности.

Важно: это не браузерный/веб-проект Ярми. Веб-проект живет отдельно. Этот репозиторий отвечает за мобильную съемку коротких реальных видео и за будущий capture-flow, на котором строится доказательство происхождения контента.

Ярми - не "соцсеть видео" и не "TikTok без фильтров". Продукт начинается как лента реальных моментов, но стратегически строит инфраструктуру доверия к происхождению цифрового контента.

Ключевая формула:

```text
Реальное событие -> подтвержденная фиксация -> публикация -> история доверия
```

## Главное Обещание

Ярми подтверждает происхождение видео: ролик был снят внутри приложения, в непрерывной camera-сессии, без импорта из галереи, склейки, фильтров и AI-обработки в capture-flow.

Нельзя формулировать обещание как абсолютную "истину" события. Пользователь может снять постановочную сцену или экран другого устройства. Ярми подтверждает provenance/capture-flow, а не философскую истинность происходящего.

Правильная формулировка:

```text
Мы подтверждаем происхождение видео и целостность capture-flow.
```

Неправильная формулировка:

```text
Мы доказываем абсолютную правду.
```

## Продуктовый Ориентир

Ярми берет у сторис скорость, присутствие и легкость: человек быстро показывает, где он находится, что делает и что происходит вокруг него прямо сейчас.

Из Telegram Stories берем:

- быстрый момент;
- легкий просмотр;
- реакции;
- контроль автора;
- ощущение живого присутствия.

От Telegram Stories отличаемся:

- видео записывается только через встроенную камеру приложения;
- загрузка из галереи запрещена;
- фильтры, AI-генерация, внешний монтаж и импортированные файлы запрещены;
- медиа-редактор не должен становиться главным продуктом;
- сторис-механики не должны появляться раньше стабильного capture-flow.

По реакции Ярми может стремиться к простоте Telegram Stories. По развлекательности Ярми может смотреть в сторону легкости TikTok, но не должен превращаться во второй TikTok и терять принцип подтвержденного происхождения.

## Capture Event

Главный будущий объект системы - не видео, а `Capture Event`.

Видео - это media asset.
`Capture Event` - доказательство того, как этот media asset появился.

Будущее событие должно фиксировать:

- id события;
- id автора;
- время записи;
- длительность;
- hash оригинального видео;
- hash опубликованной версии;
- версию приложения;
- платформу и параметры камеры;
- результат проверки непрерывности записи;
- факт отсутствия импорта из галереи;
- факт отсутствия склейки;
- факт отсутствия AI/edit pipeline;
- геолокацию, если пользователь разрешил;
- дополнительные сигналы, если пользователь разрешил;
- подпись события.

Архитектурный принцип:

```text
Content Asset
- original video
- transcoded video
- thumbnail
- playback versions

Event Proof
- hashes
- signature
- capture metadata
- verification state
- trust signals
```

Видео можно сжимать и перекодировать. Event Proof должен оставаться неизменным.

## Threat Model

Ярми защищается от:

- импорта из галереи;
- монтажа до публикации;
- AI-генерации файла внутри capture-flow;
- подмены опубликованного видео после съемки;
- потери связи между оригиналом, опубликованной версией и событием съемки.

Ярми не может полностью защититься от:

- съемки чужого экрана;
- постановочной сцены;
- spoof геолокации;
- root/jailbreak;
- модифицированного клиента;
- физической съемки поддельного объекта.

Поэтому в продукте и документации использовать термин "подтвержденное происхождение", а не "абсолютная правда".

## Основной Пользовательский Сценарий

1. Пользователь открывает приложение.
2. Камера открывается максимально быстро.
3. Пользователь снимает реальное видео одним непрерывным дублем.
4. После записи видит предпросмотр.
5. Пользователь может обрезать начало/конец.
6. Затем выбирает: опубликовать или удалить.
7. Если пользователь не авторизован, вход запрашивается только перед публикацией, чтобы момент съемки не был потерян.

## Правила MVP

- Длина видео: до 60 секунд.
- Запись: один непрерывный дубль.
- Паузы во время записи нет.
- Выбора видео из галереи нет.
- Фильтров нет.
- AI-обработки в capture-flow нет.
- Счетчики просмотров, лайки и комментарии не входят в MVP, но являются целевым направлением развития историй.
- Обрезка начала/конца разрешена: для продукта важен факт съемки через камеру, а не точная сырая длительность.
- В MVP делаем обычную запись с одной камеры.
- Dual-camera в MVP не делаем.
- Числовой публичный Verification Score в MVP не делаем.

## Verification

В MVP достаточно состояния:

```text
Снято в Ярми
```

Позже можно добавить уровни:

```text
Подтвержденная съемка
Подтвержденная съемка + дополнительные сигналы
Высокая достоверность происхождения
```

Число вроде `92 / 100` лучше оставить для внутренней системы или будущего B2B/API, пока нет объяснимой модели доверия для массового пользователя.

## Privacy

Геолокация, устройство и дополнительные сигналы - чувствительные данные.

Нужно разделять:

- публичные proof-данные;
- приватные proof-данные автора;
- серверные trust-сигналы.

Геолокация не должна быть обязательной для базовой ценности продукта.

## Архитектурное Направление

С самого начала держим проект модульным. Камеру нужно проектировать так, чтобы позже ее можно было заменить на нативную реализацию.

Рекомендуемая структура:

```text
lib/
  app/
  features/
    camera_capture/
  shared/
```

Правила:

- UI камеры и реализация камеры должны быть разделены.
- Не размазывать прямые вызовы `camera` plugin по всему приложению.
- Продуктовые сценарии складывать в feature-срезы.
- Переиспользуемые утилиты и базовые компоненты держать в `shared/`.
- На старте писать просто: не усложнять архитектуру раньше времени.
- Не смешивать media asset и future event proof в одну неуправляемую сущность.

UI стек:

- Forui: кнопки, иконки, диалоги, sheets, toasts и общая UI-система.
- Custom Yarmy UI: камера, record button, progress ring, camera overlay и trim.
- `flutter_animate`: пульсация, переходы, реакции и появление элементов.
- Forui не должен заменять кастомный camera experience.

## Правила Разработки

- Все продуктовые UI-тексты должны быть на русском.
- Первый экран приложения должен быть практическим, не лендингом.
- Целевой первый экран - камера.
- Не добавлять социальную ленту до рабочего MVP съемки.
- Не копировать Telegram Stories целиком: использовать их как продуктовый ориентир для скорости, приватности, реакций и легкого присутствия.
- Не добавлять города, геолокацию, "рядом" и фильтры в MVP.
- Не добавлять просмотры, лайки и комментарии до рабочего MVP съемки, но проектировать продукт так, чтобы их можно было добавить позже.
- Не делать dual-camera до стабильной single-camera записи.
- Не добавлять загрузку из галереи даже как debug/hidden-функцию без явного запроса. Если такая возможность когда-либо обсуждается, сначала нужно явно зафиксировать, что она противоречит принципу подтвержденного происхождения.
- Не добавлять AI/edit pipeline в capture-flow.

## Git И Ветки

- Рабочие ветки для задач называются `yarmy-N`, где `N` - следующий номер.
- Первая рабочая ветка: `yarmy-1`.
- Для следующей задачи увеличивать номер: `yarmy-2`, `yarmy-3` и так далее.
- Перед созданием новой ветки смотреть существующие ветки и выбирать следующий свободный номер.
- Не пушить изменения во внешний сервис без явного подтверждения пользователя.

Команда "Выполни дев релиз" означает: проверить изменения, запустить проверки, сделать коммит, push текущей ветки, создать PR в `main` при необходимости, замержить PR, переключиться на `main` и подтянуть `origin/main`. Перед push, созданием PR и merge PR нужно получить явное подтверждение пользователя.

## Важно Для Будущих ИИ-Чатов

При открытии этого репозитория в новом чате не считать его браузерной Ярми. Это мобильное приложение.

Стратегическая цель - не сделать второй TikTok и не сделать клон Telegram Stories. Ярми строит платформу подтвержденной реальности: реальные видеоистории, снятые внутри приложения, без загрузки из галереи и без генеративного/монтажного слоя в capture-flow.

Главная долгосрочная ценность - граф подтвержденных событий, а не лайки, просмотры или сами видеофайлы.

Перед изменением кода нужно изучить текущие файлы проекта и следовать Flutter-конвенциям, которые уже есть в репозитории.
