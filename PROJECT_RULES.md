# Правила Проекта Mobile Yarmy

## Смысл Проекта

Yarmy - это продукт про реальные видеоистории. История показывает, где пользователь находится, что делает и что происходит вокруг него прямо сейчас.

Yarmy переносит идеологию сторис в браузерный мир, но мобильное приложение отвечает за главное: настоящий контент, снятый через камеру приложения.

## Продуктовые Принципы

- История в Yarmy - это живое присутствие, а не импортированный медиафайл.
- Видео должно быть снято внутри приложения.
- Загрузка из галереи запрещена.
- AI-генерация, фильтры, внешний монтаж и импортированные файлы не входят в capture-flow.
- Пользователь должен быстро снять момент, посмотреть предпросмотр и решить: опубликовать или удалить.
- Авторизация не должна мешать съемке: вход запрашивается только перед публикацией.
- Зрители должны уметь быстро реагировать на истории.

## Ориентиры

- По философии сторис Yarmy ориентируется на Telegram Stories: быстрый момент, легкий просмотр, реакции, контроль автора.
- По развлекательной легкости Yarmy может смотреть в сторону TikTok.
- Yarmy не должен становиться клоном Telegram Stories или вторым TikTok.
- Главное отличие Yarmy - настоящий контент: в Telegram Stories можно выбрать медиа из галереи, в Yarmy нельзя.

## MVP

- Первый экран приложения - камера.
- Запись видео - один непрерывный дубль.
- Паузы во время записи нет.
- Максимальная длина видео - до 60 секунд.
- После записи пользователь видит предпросмотр.
- Пользователь может обрезать начало и конец.
- Пользователь может опубликовать, удалить или переснять историю.
- В MVP используется обычная single-camera запись.

## Не Входит В MVP

- Загрузка из галереи.
- Фильтры.
- AI-обработка.
- Внешний монтаж.
- Dual-camera.
- Социальная лента.
- Города, геолокация, "рядом".
- Просмотры, лайки и комментарии.

Просмотры, лайки, комментарии, ответы, настройки видимости, реакции и профильные подборки - важное направление развития после стабильного MVP съемки.

## Архитектурные Правила

- Flutter / Dart - основа приложения.
- Forui - выбранная UI-система проекта.
- `flutter_animate` - выбранная библиотека для UI-анимаций проекта.
- UI камеры и реализация камеры должны быть разделены.
- Прямые вызовы Flutter `camera` plugin не должны расползаться по приложению.
- Камера должна быть изолирована за абстракцией, чтобы позже заменить ее на нативную реализацию.
- Продуктовые сценарии складываются в feature-срезы.
- Переиспользуемые компоненты и утилиты живут в `shared/`.
- На старте не усложнять архитектуру раньше времени.

Рекомендуемая структура:

```text
lib/
  app/
  features/
    camera_capture/
  shared/
```

## UI Стек

- Forui: кнопки, иконки, диалоги, sheets, toasts и общая UI-система.
- Custom Yarmy UI: камера, record button, progress ring, camera overlay и trim.
- `flutter_animate`: пульсация, переходы, реакции и появление элементов.

Forui не должен заменять кастомный camera experience. Первый экран камеры проектируется как собственный Yarmy UI поверх camera preview.

## Git И Ветки

- Рабочие ветки для задач называются `yarmy-N`, где `N` - следующий номер.
- Первая рабочая ветка: `yarmy-1`.
- Для следующей задачи нужно увеличивать номер: `yarmy-2`, `yarmy-3` и так далее.
- Перед созданием новой ветки нужно посмотреть существующие ветки и выбрать следующий свободный номер.
- Не пушить изменения во внешний сервис без явного подтверждения пользователя.

## Команда "Выполни Дев Релиз"

Когда пользователь говорит "Выполни дев релиз", нужно подготовить полный dev-release flow для текущей ветки:

1. Проверить статус рабочей директории и состав изменений.
2. Запустить релевантные проверки.
3. Сделать коммит, если есть незакоммиченные изменения.
4. Запушить текущую ветку.
5. Создать PR в `main`, если PR еще нет.
6. Замержить PR.
7. Переключиться на `main`.
8. Подтянуть изменения из `origin/main`.

Перед внешними финальными действиями, которые меняют GitHub, все равно нужно получить явное подтверждение пользователя: push, создание PR и merge PR.
