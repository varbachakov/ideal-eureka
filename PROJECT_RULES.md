# Правила Проекта Mobile Yarmy

Главный продуктовый источник: `PRODUCT_CONCEPT.md`. Этот файл - короткая рабочая версия правил для разработки.

## Смысл Проекта

Ярми - мобильный capture-клиент платформы подтвержденной реальности.

Продукт начинается как приложение коротких реальных видеоисторий, но стратегически строит инфраструктуру доверия к происхождению цифрового контента.

Главный объект системы - не видео само по себе, а событие фиксации:

```text
Реальное событие -> подтвержденная фиксация -> публикация -> история доверия
```

## Главное Обещание

Ярми подтверждает происхождение видео: ролик был снят внутри приложения, в непрерывной camera-сессии, без импорта из галереи, склейки, фильтров и AI-обработки в capture-flow.

Ярми не должен обещать абсолютную "истину" события. Платформа подтверждает provenance/capture-flow, а не доказывает, что сцена не была постановочной.

## Продуктовые Принципы

- История в Ярми - это живое присутствие, а не импортированный медиафайл.
- Видео должно быть снято внутри приложения.
- Загрузка из галереи запрещена.
- AI-генерация, фильтры, внешний монтаж и импортированные файлы не входят в capture-flow.
- Обрезка начала/конца разрешена: для продукта важен факт съемки через камеру Ярми, а не каждая сырая секунда.
- Пользователь должен быстро снять момент, посмотреть предпросмотр и решить: опубликовать или удалить.
- Авторизация не должна мешать съемке: вход запрашивается только перед публикацией.
- Зрители должны уметь быстро реагировать на истории.

## Ориентиры

- По философии сторис Ярми ориентируется на Telegram Stories: быстрый момент, легкий просмотр, реакции, контроль автора.
- По развлекательной легкости Ярми может смотреть в сторону TikTok.
- Ярми не должен становиться клоном Telegram Stories или вторым TikTok.
- Главное отличие Ярми - подтвержденное происхождение: в Telegram Stories можно выбрать медиа из галереи, в Ярми нельзя.

## Threat Model

Ярми защищается от:

- импорта из галереи;
- монтажа до публикации;
- AI-генерации файла внутри capture-flow;
- подмены опубликованного видео после съемки;
- потери связи между оригиналом, опубликованной версией и событием съемки.

Ярми не может полностью защититься от:

- съемки чужого экрана;
- постановочной сцены;
- spoof геолокации;
- root/jailbreak;
- модифицированного клиента;
- физической съемки поддельного объекта.

Поэтому правильная формулировка доверия: "подтвержденное происхождение", а не "абсолютная правда".

## Capture Event

Будущая публикация должна проектироваться как `Capture Event`.

Минимальное направление данных:

- id события;
- id автора;
- время записи;
- длительность;
- hash оригинального видео;
- hash опубликованной версии;
- версия приложения;
- параметры камеры;
- состояние проверки capture-flow;
- подпись события.

Долгосрочно контент и доказательство события должны храниться отдельно:

```text
Content Asset
- original video
- transcoded video
- thumbnail
- playback versions

Event Proof
- hashes
- signature
- capture metadata
- verification state
- trust signals
```

Видео можно сжимать и перекодировать. Event Proof должен оставаться неизменным.

## Verification

В MVP достаточно простого состояния:

```text
Снято в Ярми
```

Позже можно добавить уровни:

```text
Подтвержденная съемка
Подтвержденная съемка + дополнительные сигналы
Высокая достоверность происхождения
```

Числовой `Verification Score` лучше не показывать рано в массовом интерфейсе, чтобы не создавать ложное ощущение абсолютной истины.

## Privacy

Геолокация, устройство и дополнительные сигналы - чувствительные данные.

Нужно разделять:

- публичные proof-данные;
- приватные proof-данные автора;
- серверные trust-сигналы, которые используются для проверки, но не публикуются напрямую.

Геолокация не должна быть обязательной для базовой ценности продукта.

## MVP

- Первый экран приложения - камера.
- Запись видео - один непрерывный дубль.
- Паузы во время записи нет.
- Максимальная длина видео - до 60 секунд.
- После записи пользователь видит предпросмотр.
- Пользователь может обрезать начало и конец.
- Пользователь может опубликовать, удалить или переснять историю.
- В MVP используется обычная single-camera запись.

## Не Входит В MVP

- Загрузка из галереи.
- Фильтры.
- AI-обработка.
- Внешний монтаж.
- Dual-camera.
- Социальная лента до стабильного capture-flow.
- Города, геолокация, "рядом".
- Просмотры, лайки и комментарии.
- Числовой публичный Verification Score.

Просмотры, лайки, комментарии, ответы, настройки видимости, реакции и профильные подборки - важное направление развития после стабильного MVP съемки.

## Архитектурные Правила

- Flutter / Dart - основа приложения.
- Forui - выбранная UI-система проекта.
- `flutter_animate` - выбранная библиотека для UI-анимаций проекта.
- UI камеры и реализация камеры должны быть разделены.
- Прямые вызовы Flutter `camera` plugin не должны расползаться по приложению.
- Камера должна быть изолирована за абстракцией, чтобы позже заменить ее на нативную реализацию.
- Продуктовые сценарии складываются в feature-срезы.
- Переиспользуемые компоненты и утилиты живут в `shared/`.
- На старте не усложнять архитектуру раньше времени.

Рекомендуемая структура:

```text
lib/
  app/
  features/
    camera_capture/
  shared/
```

## UI Стек

- Forui: кнопки, иконки, диалоги, sheets, toasts и общая UI-система.
- Custom Yarmy UI: камера, record button, progress ring, camera overlay и trim.
- `flutter_animate`: пульсация, переходы, реакции и появление элементов.

Forui не должен заменять кастомный camera experience. Первый экран камеры проектируется как собственный Yarmy UI поверх camera preview.

## Git И Ветки

- Рабочие ветки для задач называются `yarmy-N`, где `N` - следующий номер.
- Первая рабочая ветка: `yarmy-1`.
- Для следующей задачи нужно увеличивать номер: `yarmy-2`, `yarmy-3` и так далее.
- Перед созданием новой ветки нужно посмотреть существующие ветки и выбрать следующий свободный номер.
- Не пушить изменения во внешний сервис без явного подтверждения пользователя.

## Команда "Выполни Дев Релиз"

Когда пользователь говорит "Выполни дев релиз", нужно подготовить полный dev-release flow для текущей ветки:

1. Проверить статус рабочей директории и состав изменений.
2. Запустить релевантные проверки.
3. Сделать коммит, если есть незакоммиченные изменения.
4. Запушить текущую ветку.
5. Создать PR в `main`, если PR еще нет.
6. Замержить PR.
7. Переключиться на `main`.
8. Подтянуть изменения из `origin/main`.

Перед внешними финальными действиями, которые меняют GitHub, все равно нужно получить явное подтверждение пользователя: push, создание PR и merge PR.
